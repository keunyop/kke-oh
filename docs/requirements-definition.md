# KKE-OH 현재 구현 기준 요구사항 정의서

## 1. 문서 목적

이 문서는 `kke-oh` 코드베이스를 기준으로 현재까지 구현된 제품 요구사항을 정리한 문서다.  
기획안이 아니라 실제 소스코드, 라우트, 데이터 구조, 테스트 코드에서 확인되는 범위만 반영했다.

- 기준일: 2026-03-16
- 분석 기준: `app/`, `components/`, `lib/`, `supabase/migrations/`, `data/`, 테스트 파일
- 해석 원칙: 코드에 명시된 동작을 우선하고, 일부는 구현 흐름을 바탕으로 추론했다

## 2. 서비스 개요

KKE-OH는 아이들이 쉽게 만들고 플레이할 수 있는 HTML 게임 놀이터다.  
사용자는 로그인 후 직접 만든 HTML/ZIP 게임을 업로드하거나 AI로 새 게임을 생성할 수 있다.  
생성된 게임은 기본적으로 초안으로 저장되며, 작성자가 테스트 후 공개할 수 있다.

## 3. 사용자 유형

### 3.1 비로그인 사용자

- 공개된 게임 목록을 조회할 수 있어야 한다.
- 공개된 게임 상세 페이지에서 게임을 플레이할 수 있어야 한다.
- 좋아요/싫어요, 피드백, 신고를 보낼 수 있어야 한다.
- 리더보드가 활성화된 게임에서는 점수 등록 시 표시 이름을 직접 입력할 수 있어야 한다.

### 3.2 일반 로그인 사용자

- ID/비밀번호 기반으로 회원가입 및 로그인할 수 있어야 한다.
- 게임을 업로드하거나 AI로 생성할 수 있어야 한다.
- 자신의 게임 초안을 조회, 수정, 공개, 비공개 전환, 삭제할 수 있어야 한다.
- 자신의 비공개 초안/숨김 게임을 미리보기할 수 있어야 한다.

### 3.3 관리자

- 관리자 계정만 관리자 화면에 접근할 수 있어야 한다.
- 전체 게임 목록을 조회하고 숨김/복구/삭제 조치를 할 수 있어야 한다.
- 신고, 숨김, 삭제, CDN allowlist 위반 여부를 한 화면에서 확인할 수 있어야 한다.

## 4. 핵심 기능 요구사항

### 4.1 홈 화면

- 공개 상태이며 숨김 처리되지 않았고 신고 누적 2회 미만인 게임만 노출해야 한다.
- 게임 카드는 다음 정보를 보여줘야 한다.
  - 썸네일
  - 게임명
  - 작성자명
  - 플레이 수
  - 반응 정보
- 검색어(`q`)로 게임 제목, 설명, 카테고리, 작성자 기준 필터링이 가능해야 한다.
- 리더보드가 활성화된 게임 중 최고 점수 보유자를 별도 챔피언 영역으로 노출해야 한다.
- 게임은 인기 점수 기반으로 정렬되어야 한다.

### 4.2 다국어 지원

- 서비스는 한국어(`ko`)와 영어(`en`)를 지원해야 한다.
- 언어는 쿠키(`kkeoh_locale`) 또는 국가/브라우저 헤더를 기준으로 결정되어야 한다.
- 기본 언어는 한국 사용자는 `ko`, 그 외는 `en`이어야 한다.

### 4.3 인증

- 회원가입과 로그인은 ID + 비밀번호만으로 처리되어야 한다.
- ID는 정규화되어 중복 검사를 해야 한다.
- 세션은 쿠키(`kkeoh_session`) 기반으로 관리되어야 한다.
- 세션 유지 기간은 30일이어야 한다.
- Supabase 인증 테이블이 없을 때는 filesystem 인증 저장소로 fallback 가능해야 한다.

### 4.4 게임 생성: AI 생성

- 로그인한 사용자만 AI 생성 기능을 사용할 수 있어야 한다.
- 프롬프트는 8자 이상 1200자 이하로 제한해야 한다.
- 표시용 게임명, URL 게임명, 설명, 썸네일은 선택 입력 가능해야 한다.
- 표시용 게임명을 직접 입력한 경우 URL 게임명도 함께 입력되어야 한다.
- URL 게임명은 영문 소문자, 숫자, 하이픈 조합만 허용해야 한다.
- AI가 반환한 HTML은 단일 독립 실행형 문서여야 한다.
- AI가 만든 게임은 초안으로 저장되어야 한다.
- AI가 만든 게임은 기본적으로 리더보드를 활성화해야 한다.
- 썸네일을 직접 업로드하지 않으면 AI 썸네일 또는 플레이스홀더를 사용해야 한다.

### 4.5 게임 생성: 직접 업로드

- 로그인한 사용자만 직접 업로드 기능을 사용할 수 있어야 한다.
- 업로드 방식은 다음 두 가지를 제공해야 한다.
  - 단일 HTML 업로드
  - ZIP 업로드

#### 단일 HTML 업로드

- 표시용 게임명, URL 게임명, 설명은 필수여야 한다.
- HTML 파일 또는 HTML 문자열이 필요해야 한다.
- 썸네일 이미지는 선택 입력이어야 한다.
- 업로드된 게임은 초안으로 저장되어야 한다.
- 직접 업로드 게임은 기본적으로 리더보드를 비활성화해야 한다.

#### ZIP 업로드

- ZIP 업로드 전 검사 단계가 있어야 한다.
- ZIP 파일은 다음 제약을 만족해야 한다.
  - 전체 15MB 이하
  - 파일당 5MB 이하
  - 총 파일 수 200개 이하
  - 최소 1개의 HTML 파일 포함
  - 허용 확장자만 포함
- ZIP 안의 `game.json`은 사용자 업로드 시 무시되어야 한다.
- 시작 파일은 `index.html` 우선, 없으면 첫 번째 HTML 파일이어야 한다.
- ZIP 검사 결과로 다음 정보를 사용자에게 제공해야 한다.
  - 시작 파일 경로
  - HTML 파일 수
  - 썸네일 후보 수
  - 외부 링크 allowlist 위반 여부
- ZIP 검사 세션은 서버 메모리에 임시 보관되며 만료될 수 있어야 한다.
- 만료 시 사용자는 ZIP을 다시 업로드해야 한다.

### 4.6 URL 게임명 관리

- 게임 상세 URL은 `/game/{slug}` 형식이어야 한다.
- slug는 중복되면 안 된다.
- filesystem 모드와 supabase 모드 모두 slug 중복 확인이 가능해야 한다.
- AI 생성 시 slug를 직접 입력하지 않으면 자동 생성해야 한다.
- 수동 입력한 slug가 이미 사용 중이면 저장을 막아야 한다.

### 4.7 썸네일 처리

- 썸네일은 PNG/JPG/JPEG/WEBP 업로드를 허용해야 한다.
- 썸네일은 업로드 시 WEBP로 최적화되어야 한다.
- 최대 크기는 1200x825 안쪽으로 리사이즈되어야 한다.
- 썸네일이 없으면 게임명 기반 플레이스홀더 이미지를 생성해야 한다.

### 4.8 게임 상세/플레이

- 공개 게임은 누구나 접근할 수 있어야 한다.
- 초안 또는 숨김 게임은 작성자만 접근할 수 있어야 한다.
- 삭제된 게임은 접근할 수 없어야 한다.
- 게임은 iframe 안에서 실행되어야 한다.
- iframe은 스크립트 실행, same-origin, pointer lock만 허용해야 한다.
- 전체화면 진입 버튼을 제공해야 한다.
- 공개 게임 상세에서는 좋아요/싫어요/피드백 기능을 제공해야 한다.
- 초안 미리보기에서는 공개 반응 기능 대신 테스트 안내를 보여줘야 한다.

### 4.9 플레이 수 집계

- 공개 게임 상세 진입 시 플레이 수 집계를 시도해야 한다.
- 동일 브라우저 세션 기준 5분 이내 중복 집계는 막아야 한다.
- 비공개 초안/숨김 미리보기에서는 공개 플레이 집계를 하지 않아야 한다.

### 4.10 반응, 피드백, 신고

#### 반응

- 사용자는 좋아요 또는 싫어요 중 하나를 남길 수 있어야 한다.
- 반응 상태는 브라우저 쿠키로 기억되어야 한다.
- 이전 반응을 바꾸면 카운트가 조정되어야 한다.

#### 피드백

- 공개 게임에 대해 자유 텍스트 피드백을 보낼 수 있어야 한다.
- 메시지가 비어 있으면 전송할 수 없어야 한다.

#### 신고

- 게임 신고 사유를 서버에 저장할 수 있어야 한다.
- 신고 누적 2회 이상이면 자동 숨김 처리되어야 한다.
- 자동 숨김 시 숨김 사유를 기록해야 한다.

### 4.11 리더보드

- 리더보드가 활성화된 게임만 점수 등록을 허용해야 한다.
- 점수는 1 이상 999,999,999 이하 정수만 허용해야 한다.
- 플레이어 이름은 공백 정리 후 최대 24자까지 허용해야 한다.
- 로그인 사용자는 자신의 로그인 ID로 점수를 저장해야 한다.
- 비로그인 사용자는 이름을 입력해야 하며, 브라우저에 저장 후 재사용 가능해야 한다.
- 같은 이름과 같은 점수가 15초 이내 중복 제출되면 재저장하지 않아야 한다.
- 게임이 종료되거나 승패가 확정될 때 점수를 자동 수집할 수 있어야 한다.
- 초안 미리보기에서도 작성자는 점수 저장을 테스트할 수 있어야 한다.
- 홈 화면에서는 게임별 최고 점수 보유자를 챔피언으로 노출해야 한다.

### 4.12 내 게임 관리

- 로그인 사용자는 자신의 게임 목록만 볼 수 있어야 한다.
- 목록에서 다음 작업을 할 수 있어야 한다.
  - 미리보기/열기
  - 수정
  - 게시
  - 초안 전환
  - 삭제
- 삭제는 실제 파일 제거가 아니라 상태를 `REMOVED`로 바꾸는 소프트 삭제여야 한다.

### 4.13 게임 수정

- 작성자만 자신의 게임을 수정할 수 있어야 한다.
- 수정 모드는 다음 네 가지를 지원해야 한다.
  - 상세 정보만 수정
  - HTML로 교체
  - ZIP으로 교체
  - AI로 다시 생성
- 제목, 설명, 썸네일은 수정 가능해야 한다.
- AI 재생성 시 현재 게임 정보와 기존 HTML 일부를 프롬프트 문맥으로 넘겨야 한다.
- AI 재생성 모드에서는 리더보드를 활성화해야 한다.

### 4.14 관리자 기능

- 관리자 계정은 현재 고정된 로그인 ID 목록으로 판정해야 한다.
- 관리자 페이지는 로그인 후 관리자 계정만 접근 가능해야 한다.
- 관리자 화면에서는 다음 기능을 제공해야 한다.
  - 전체 게임 목록 조회
  - 검색
  - 빠른 필터
    - 전체
    - 공개
    - 숨김
    - 신고
    - 플래그
    - 삭제됨
  - 숨김
  - 숨김 해제
  - 삭제
  - 새로고침
- 관리자 화면에서 다음 정보를 보여줘야 한다.
  - 제목, 설명
  - 작성자
  - 게임 ID
  - 생성/수정일
  - 신고 수
  - 좋아요/싫어요
  - 플레이 수
  - 숨김 사유
  - allowlist 위반 여부

## 5. 비기능 요구사항

### 5.1 저장소/운영 모드

- 서비스는 `filesystem` 모드와 `supabase` 모드를 지원해야 한다.
- 환경변수에 따라 저장 드라이버를 자동 선택할 수 있어야 한다.
- filesystem 모드에서는 로컬 디렉터리 구조로 게임과 인증 정보를 저장해야 한다.
- supabase 모드에서는 메타데이터는 Supabase, 게임 자산은 R2에 저장해야 한다.

### 5.2 보안

- 게임 자산 경로는 상위 디렉터리 접근을 허용하면 안 된다.
- HTML 게임 응답에는 제한된 CSP를 적용해야 한다.
- 외부 URL은 허용된 CDN 도메인만 정상 범위로 간주해야 한다.
- 세션/반응/플레이 쿠키는 `httpOnly`, `sameSite=lax` 정책을 사용해야 한다.

### 5.3 성능/운영

- 홈, 게임 상세, 내 게임, 관리자 화면은 동적 렌더링을 허용해야 한다.
- 게임 생성/수정/관리 작업 후 관련 경로는 재검증되어야 한다.

## 6. 현재 구현 기준 제약 및 확인사항

### 6.1 구현되어 있으나 현재 연결되지 않은 요소

- Cloudflare Turnstile 검증 유틸은 존재하지만 현재 업로드/인증 흐름에 연결되어 있지 않다.
- 업로드 횟수 제한(`upload_events` 기반)과 blocklist 조회 유틸이 존재하지만 현재 업로드 라우트에서 호출되지 않는다.
- `ADMIN_SECRET` 환경변수는 예시 파일에 남아 있지만 현재 관리자 인증 판단에는 사용되지 않는다.
- `/api/admin/auth`와 `/api/admin/blocklist`는 실사용 기능이 아니라 비활성 응답을 반환한다.

### 6.2 현재 코드에서 드러나는 운영상 특이사항

- 관리자 계정은 설정 화면 없이 코드 상의 고정 ID 두 개로 관리된다.
- 한국어 문구 일부는 소스에 문자 인코딩 깨짐이 포함되어 있어 실제 UI 품질 이슈 가능성이 있다.
- 수동 업로드/수동 수정 화면에는 리더보드 on/off 설정 UI가 없다.
- 플레이 수 증가는 Supabase RPC를 사용하지만 현재 IP 해시는 빈 문자열로 전달된다.
- AI 생성 API는 `generate`와 `generate-v2` 두 버전이 공존하며, 실제 화면은 `generate-v2`를 사용한다.

## 7. 요구사항 우선 정리

현재 구현 기준 제품의 핵심 요구사항은 다음 세 가지로 요약된다.

1. 아이들이 쉽게 게임을 만들 수 있는 간단한 제작 경험 제공
2. 초안-테스트-게시 흐름을 통해 안전하게 공개할 수 있는 운영 구조 제공
3. 공개 게임 플레이, 반응, 리더보드, 관리자 검수를 포함한 커뮤니티 운영 기능 제공
