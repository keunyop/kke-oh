# KKE-OH 요구사항 정의서

## 1. 문서 목적

이 문서는 `kke-oh`의 제품 요구사항을 정리한 문서다.  
현재 코드 분석, 기존 문서, 추가 요구사항을 반영해 앞으로 맞춰야 할 목표 요구사항 기준으로 정리한다.

## 2. 서비스 개요

KKE-OH는 아이들이 쉽게 만들고 플레이할 수 있는 HTML 게임 놀이터다.  
사용자는 로그인 후 직접 만든 HTML/ZIP 게임을 업로드하거나 AI로 새 게임을 생성·수정할 수 있다.  
서비스는 게임 플레이와 제작 경험뿐 아니라 포인트, 광고, 리더보드, 관리자 검수까지 포함하는 운영형 플랫폼으로 확장되어야 한다.

## 3. 사용자 유형

### 3.1 비로그인 사용자

- 공개된 게임 목록을 조회할 수 있어야 한다.
- 공개된 게임 상세 페이지에서 게임을 플레이할 수 있어야 한다.
- 좋아요/싫어요, 피드백, 신고를 보낼 수 있어야 한다.
- 리더보드가 활성화된 게임에서는 점수 등록 시 표시 이름을 직접 입력할 수 있어야 한다.
- 광고 노출은 가능하되, 아동 대상 서비스 정책에 맞는 안전한 광고만 보여야 한다.

### 3.2 일반 로그인 사용자

- ID/비밀번호 기반으로 회원가입 및 로그인할 수 있어야 한다.
- 게임을 업로드하거나 AI로 생성·수정할 수 있어야 한다.
- 자신의 게임 초안, 게시, 숨김 상태를 조회하고 수정, 게시, 게시 숨김, 삭제할 수 있어야 한다.
- 자신의 비공개 초안/숨김 게임을 미리보기할 수 있어야 한다.
- 포인트 잔액과 사용/획득 내역을 확인할 수 있어야 한다.
- 게임 플레이, 리워드 광고 시청, 포인트 구매를 통해 포인트를 획득할 수 있어야 한다.

### 3.3 관리자

- 관리자 계정만 관리자 화면에 접근할 수 있어야 한다.
- 전체 게임 목록을 조회하고 숨김/복구/삭제 조치를 할 수 있어야 한다.
- 신고, 숨김, 삭제, 리더보드 사용 여부, allowlist 위반 여부를 한 화면에서 확인할 수 있어야 한다.
- 신고가 누적된 게임은 별도 alert 형태로 우선 검토 대상으로 보여야 한다.
- 포인트 정책, AI 모델 정책, 광고 운영 상태를 관리하거나 점검할 수 있는 기반이 필요하다.

## 4. 핵심 기능 요구사항

### 4.1 홈 화면

- 공개 상태이며 숨김 처리되지 않은 게임만 노출해야 한다.
- 신고 누적만으로 홈 노출을 자동 제한하지 않아야 한다.
- 신고가 2회 이상 누적된 게임은 관리자 화면에서 alert 또는 강조 상태로 보여야 한다.
- 게임 카드는 다음 정보를 보여줘야 한다.
  - 썸네일
  - 게임명
  - 작성자명
  - 플레이 수
  - 반응 정보
- 헤더에서는 별도 포인트 버튼을 노출하지 않아야 하며, 포인트 진입은 아바타 메뉴 또는 전용 페이지에서 처리해야 한다.
- 헤더 반응형 레이아웃에서는 `Create Game` 버튼의 폭과 높이를 유지하고 검색창이 먼저 줄어들어야 하며, `Create Game` 텍스트가 2줄로 줄바꿈되면 안 된다.
- 홈 검색은 입력 즉시 결과가 바뀌는 filtering 방식이어야 하며, 별도 제출 없이 현재 화면의 게임 목록을 갱신해야 한다.
- 홈 검색 대상은 게임명과 작성자명이어야 한다.
- 게임 카드의 게임명은 1줄 고정이어야 하며 카드 폭을 넘기면 `...`으로 말줄임 처리하고 mouse over 시 전체 이름을 확인할 수 있어야 한다.
- 게임 카드의 작성자 표시는 `Creator` 레이블 없이 이름만 보여줘야 하며, 길면 1줄 말줄임 처리하고 mouse over 시 전체 이름을 확인할 수 있어야 한다.
- 홈 화면의 리더보드 챔피언 영역은 메인 콘텐츠 상단부로 올려 더 먼저 보이도록 구성해야 한다.
- 홈 리더보드 챔피언 영역은 전체 사이트 톤과 어울리는 블랙보드 느낌의 디자인으로 개선해야 한다.
- 홈 리더보드 챔피언 영역의 높이는 현재보다 낮춰 첫 화면에서 게임 카드와 함께 더 compact 하게 보이도록 해야 한다.
- 홈 리더보드 챔피언 영역은 자동으로 좌우 이동하되, 마우스 드래그와 터치 드래그로도 직접 좌우 이동할 수 있어야 한다.
- 리더보드가 활성화된 게임 중 최고 점수 보유자를 별도 챔피언 영역으로 노출해야 한다.
- 게임은 인기 점수 기반으로 정렬되어야 한다.
- 홈 및 게임 탐색 화면에는 사이트 운영 수익화를 위한 Google 광고 슬롯을 배치할 수 있어야 한다.

### 4.2 다국어 지원

- 서비스는 한국어(`ko`)와 영어(`en`)를 지원해야 한다.
- 언어는 쿠키(`kkeoh_locale`) 또는 국가/브라우저 헤더를 기준으로 결정되어야 한다.
- 기본 언어는 한국 사용자는 `ko`, 그 외는 `en`이어야 한다.
- 한국어 문구가 깨지지 않고 정상 표시되어야 한다.
- 선택한 언어가 영어일 때 회원가입 중복 ID 오류를 포함한 인증 오류 문구도 영어로 일관되게 표시되어야 한다.

### 4.3 인증 및 계정

- 회원가입과 로그인은 ID + 비밀번호만으로 처리되어야 한다.
- ID는 정규화되어 중복 검사를 해야 한다.
- 세션은 쿠키(`kkeoh_session`) 기반으로 관리되어야 한다.
- 세션 유지 기간은 30일이어야 한다.
- 관리자 권한은 허용된 관리자 계정 기준으로 판정되어야 한다.
- 로그인 사용자는 현재 포인트 잔액을 주요 화면에서 확인할 수 있어야 한다.
- 로그인 화면은 설명용 좌측 카드를 제거하고 인증 카드만 가운데에 배치해야 한다.
- 로그인/회원가입 카드에서는 `Login` 뱃지와 `You only need an ID and password.` 같은 보조 문구를 제거하고, 실제 입력과 전환에 필요한 정보만 보여야 한다.
- 회원가입 중복 ID 오류를 포함한 인증 오류는 붉은색 오류 스타일로 표시되어야 한다.

### 4.4 AI로 게임 만들기

- 로그인한 사용자만 AI 생성 기능을 사용할 수 있어야 한다.
- 프롬프트는 8자 이상 1200자 이하로 제한해야 한다.
- 표시용 게임명, URL 게임명, 설명, 썸네일은 선택 입력 가능해야 한다.
- 표시용 게임명을 직접 입력한 경우 URL 게임명도 함께 입력되어야 한다.
- URL 게임명은 영문 소문자, 숫자, 하이픈 조합만 허용해야 한다.
- 사용자는 허용된 AI 모델 목록 중 하나를 선택할 수 있어야 한다.
- 선택한 AI 모델에 따라 필요한 포인트 수가 달라져야 한다.
- 사용자는 생성 전에 선택 모델, 예상 포인트 차감량, 현재 잔액을 확인할 수 있어야 한다.
- 포인트가 부족하면 AI 생성 요청을 보낼 수 없어야 한다.
- AI가 반환한 HTML은 단일 독립 실행형 문서여야 한다.
- AI가 만든 게임은 초안으로 저장되어야 한다.
- AI가 만든 게임은 기본적으로 리더보드를 활성화해야 한다.
- 썸네일을 직접 업로드하지 않으면 AI 썸네일을 사용하고, AI 썸네일도 없을 경우 KKE-OH 로고 기반 플레이스홀더를 사용해야 한다.
- AI 생성 중에는 단계형 진행상황 UI를 보여줘야 한다.
- `How it works` 안내 문구는 현재 생성 화면 구조에 맞춰 다시 작성해야 하며, 초등학생도 바로 이해할 수 있는 쉬운 문장이어야 한다.
- AI 모델 선택 영역에서는 각 모델 설명을 초등학생 눈높이로 제공해야 한다.
- AI 모델 선택 우측의 `Current points`, `Expected cost` 요약 카드는 더 읽기 쉽고 시각적으로 정리된 디자인으로 개선해야 한다.
- AI 모델 선택 아래의 선택 입력 정보 영역은 `고급 설정`으로 접어 두고 기본값은 collapsed 상태여야 한다.

### 4.5 직접 업로드로 게임 만들기

- 로그인한 사용자만 직접 업로드 기능을 사용할 수 있어야 한다.
- 업로드 방식은 다음 두 가지를 제공해야 한다.
  - 단일 HTML 업로드
  - ZIP 업로드
- 직접 업로드한 게임도 KKE-OH 리더보드를 사용할 수 있는 방안을 제공해야 한다.

#### 단일 HTML 업로드

- 표시용 게임명과 URL 게임명은 필수여야 한다.
- 설명은 선택 입력이어야 한다.
- HTML 파일 업로드만 허용해야 한다.
- 화면에서 HTML 문자열을 직접 입력받지 않아야 한다.
- 썸네일 이미지는 선택 입력이어야 한다.
- 업로드된 게임은 초안으로 저장되어야 한다.
- 사용자는 리더보드 사용 여부를 선택할 수 있어야 한다.

#### ZIP 업로드

- ZIP 업로드 전 검사 단계가 있어야 한다.
- 표시용 게임명과 URL 게임명은 필수여야 한다.
- 설명은 선택 입력이어야 한다.
- 사용자는 리더보드 사용 여부를 선택할 수 있어야 한다.
- ZIP 업로드 모드에서는 별도의 외부 `Thumbnail image` 입력 필드를 요구하지 않아야 한다.
- ZIP 파일은 다음 제약을 만족해야 한다.
  - 전체 15MB 이하
  - 파일당 5MB 이하
  - 총 파일 수 200개 이하
  - 최소 1개의 HTML 파일 포함
  - 허용 확장자만 포함
- ZIP 안의 `game.json`은 사용자 업로드 시 무시되어야 한다.
- 시작 파일은 `index.html` 우선, 없으면 첫 번째 HTML 파일이어야 한다.
- ZIP 검사 결과로 다음 정보를 사용자에게 제공해야 한다.
  - 시작 파일 경로
  - HTML 파일 수
  - 썸네일 후보 수
  - 외부 링크 allowlist 위반 여부
- ZIP 검사 세션은 서버 메모리에 임시 보관되며 만료될 수 있어야 한다.
- 만료 시 사용자는 ZIP을 다시 업로드해야 한다.

### 4.6 AI로 게임 수정

- 작성자만 자신의 게임을 AI로 수정할 수 있어야 한다.
- AI 수정 시 현재 게임 정보와 기존 HTML 일부를 프롬프트 문맥으로 넘겨야 한다.
- 사용자는 AI 수정 시에도 허용된 AI 모델 목록 중 하나를 선택할 수 있어야 한다.
- 선택한 AI 모델에 따라 필요한 포인트 수가 달라져야 한다.
- 사용자는 수정 전에 선택 모델, 예상 포인트 차감량, 현재 잔액을 확인할 수 있어야 한다.
- 포인트가 부족하면 AI 수정 요청을 보낼 수 없어야 한다.
- AI 수정 중에는 AI 생성과 동일한 수준의 단계형 진행상황 UI를 보여줘야 한다.
- AI 수정 모드에서는 리더보드를 활성화할 수 있어야 한다.
- AI 수정 화면도 생성 화면과 같은 기준으로 모델 설명을 초등학생 눈높이로 제공해야 한다.
- AI 수정 화면의 포인트 요약 카드는 생성 화면과 동일한 디자인 체계로 정리해야 한다.
- AI 수정 화면의 선택 입력 영역도 `고급 설정`으로 기본 접힘 상태를 사용해야 한다.

### 4.7 일반 게임 수정

- 작성자만 자신의 게임을 수정할 수 있어야 한다.
- 수정 모드는 다음 세 가지를 지원해야 한다.
  - HTML로 수정
  - ZIP으로 수정
  - AI로 수정
- 제목, 설명, 썸네일은 각 수정 모드 안에서 함께 수정 가능해야 한다.
- 상세 정보만 따로 수정하는 별도 탭은 필요하지 않다.
- HTML/ZIP 수정 모드에서도 리더보드 사용 여부를 변경할 수 있어야 한다.
- 수정 화면에서는 게임명과 설명 입력을 수정 방식 선택 버튼보다 위에 배치해야 한다.
- 수정 방식 버튼 순서는 `Edit with AI`가 가장 앞에 와야 한다.
- 수정 화면 하단 버튼은 `Save changes`와 `취소`만 제공해야 한다.

### 4.8 URL 게임명 관리

- 게임 상세 URL은 `/game/{slug}` 형식이어야 한다.
- slug는 중복되면 안 된다.
- Supabase 기준으로 slug 중복 확인이 가능해야 한다.
- AI 생성 시 slug를 직접 입력하지 않으면 자동 생성해야 한다.
- 수동 입력한 slug가 이미 사용 중이면 저장을 막아야 한다.

### 4.9 썸네일 처리

- 썸네일은 PNG/JPG/JPEG/WEBP 업로드를 허용해야 한다.
- 썸네일은 업로드 시 WEBP로 최적화되어야 한다.
- 최대 크기는 1200x825 안쪽으로 리사이즈되어야 한다.
- 기본 플레이스홀더는 게임명 기반 색상 조합이 아니라 KKE-OH 로고 기반 디자인을 사용해야 한다.

### 4.10 게임 상세/플레이

- 공개 게임은 누구나 접근할 수 있어야 한다.
- 초안 또는 숨김 게임은 작성자만 접근할 수 있어야 한다.
- 삭제된 게임은 접근할 수 없어야 한다.
- 게임은 iframe 안에서 실행되어야 한다.
- iframe은 스크립트 실행, same-origin, pointer lock만 허용해야 한다.
- 전체화면 진입 버튼을 제공해야 한다.
- 공개 게임 상세에서는 좋아요/싫어요/피드백/신고 기능을 제공해야 한다.
- 좋아요, 싫어요, 피드백, 신고 버튼에는 mouse over 시 각각의 기능명이 알맞게 보이는 tooltip 또는 동등한 안내가 있어야 한다.
- 초안 미리보기에서는 공개 반응 기능 대신 테스트 안내를 보여줘야 한다.
- 게임 플레이가 끝난 뒤에는 리워드형 광고를 시청할 수 있는 진입 UI를 보여줘야 한다.
- 리워드 광고는 자동 재생이 아니라 사용자의 명시적 선택 후에만 표시되어야 한다.
- 지원되지 않는 환경에서는 리워드 광고 대신 대체 안내를 제공해야 한다.

### 4.11 반응, 피드백, 신고

#### 반응

- 사용자는 좋아요 또는 싫어요 중 하나를 남길 수 있어야 한다.
- 반응 상태는 브라우저 쿠키로 기억되어야 한다.
- 이전 반응을 바꾸면 카운트가 조정되어야 한다.
- 새로고침 후에도 현재 브라우저의 반응 상태가 UI에 반영되어야 한다.

#### 피드백

- 공개 게임에 대해 자유 텍스트 피드백을 보낼 수 있어야 한다.
- 메시지가 비어 있으면 전송할 수 없어야 한다.

#### 신고

- 게임 상세 페이지에 신고 버튼이 있어야 한다.
- 게임 신고 사유를 서버에 저장할 수 있어야 한다.
- 신고가 2회 이상 누적되면 관리자 화면에서 alert 또는 강조 상태로 보여야 한다.
- 신고 누적만으로 자동 숨김 처리하지 않아야 한다.

### 4.12 리더보드

- 리더보드가 활성화된 게임만 점수 등록을 허용해야 한다.
- AI 생성 게임과 직접 업로드 게임 모두 리더보드를 사용할 수 있어야 한다.
- 점수는 1 이상 999,999,999 이하 정수만 허용해야 한다.
- 플레이어 이름은 공백 정리 후 최대 24자까지 허용해야 한다.
- 로그인 사용자는 자신의 로그인 ID로 점수를 저장해야 한다.
- 비로그인 사용자는 이름을 입력해야 하며, 브라우저에 저장 후 재사용 가능해야 한다.
- 같은 이름과 같은 점수가 15초 이내 중복 제출되면 재저장하지 않아야 한다.
- 게임이 종료되거나 승패가 확정될 때 점수를 자동 수집할 수 있어야 한다.
- 초안 미리보기에서도 작성자는 점수 저장을 테스트할 수 있어야 한다.
- 홈 화면에서는 게임별 최고 점수 보유자를 챔피언으로 노출해야 한다.
- 사용자가 만든 게임이 리더보드를 쉽게 연동할 수 있도록 KKE-OH 표준 점수 제출 방식이 제공되어야 한다.

### 4.13 포인트 시스템

- 서비스 내에서 모으고 사용할 수 있는 포인트 개념이 있어야 한다.
- 포인트는 서버 기준의 잔액과 거래내역으로 관리되어야 한다.
- 포인트 획득 경로는 다음을 포함해야 한다.
  - 공개 게임 플레이
  - 리워드 광고 시청 완료
  - 포인트 구매
- 포인트 사용 경로는 다음을 포함해야 한다.
  - AI로 게임 만들기
  - AI로 게임 수정하기
- 선택한 AI 모델에 따라 포인트 차감량이 달라져야 한다.
- 포인트 지급과 차감은 이력으로 남아야 한다.
- 초안 미리보기나 관리자 테스트는 일반 플레이 포인트 지급 대상에서 제외할 수 있어야 한다.
- 포인트 부족 시 AI 생성/수정은 차단되어야 한다.
- 포인트 획득/차감 결과는 사용자에게 명확히 보여야 한다.

### 4.14 포인트 구매

- 로그인 사용자는 포인트를 구매할 수 있어야 한다.
- 구매 가능한 포인트 상품 목록이 있어야 한다.
- 구매 요청, 승인, 실패, 취소 상태를 구분해 관리해야 한다.
- 포인트는 결제 승인 이후에만 지급되어야 한다.
- 결제·구매 기능은 아동 대상 서비스 특성을 고려한 보호자/정책 검토가 필요하다.
- 포인트 구매 카드는 데스크톱 기준 한 줄에 3개가 정렬되도록 카드 크기와 그리드를 조정해야 한다.

### 4.15 광고 및 수익화

- KKE-OH 사이트에는 Google 광고를 배치할 수 있어야 한다.
- 광고는 홈, 탐색, 상세, 내 게임 등 사이트 영역에 배치하되 게임 조작을 방해하지 않아야 한다.
- 아동 대상 서비스 특성상 광고 요청은 child-directed 처리 기준을 따라야 한다.
- 아동 대상 또는 13세 미만 이용자 활동을 기반으로 개인화 광고를 사용하지 않아야 한다.
- 리워드형 광고는 보상 제공 광고 정책에 맞는 opt-in 흐름이어야 한다.
- 리워드 포인트는 광고 시청이 실제로 승인된 뒤에만 지급되어야 한다.
- 광고 미지원 환경이나 광고 미충족 시에도 게임 경험이 막히지 않아야 한다.

### 4.16 내 게임 관리

- 로그인 사용자는 자신의 게임 목록만 볼 수 있어야 한다.
- 목록에서 다음 작업을 할 수 있어야 한다.
  - 미리보기/열기
  - 수정
  - 게시
  - 게시 숨기기
  - 삭제
- 한번 게시한 게임은 내 게임 화면에서 `Move to Draft` 대신 공개 목록에서 숨기는 동작을 제공해야 하며, 숨김은 초안 회귀가 아니라 공개 노출만 중단하는 방식이어야 한다.
- 삭제는 실제 파일 제거가 아니라 상태를 `REMOVED`로 바꾸는 소프트 삭제여야 한다.
- 내 게임 화면의 삭제 버튼은 관리자 화면과 동일한 붉은색 danger 스타일을 사용해야 한다.
- AI 생성/수정 결과가 완료되면 사용자는 결과 화면 또는 이동 후 상태 메시지로 완료 사실을 인지할 수 있어야 한다.

### 4.17 관리자 기능

- 관리자 계정은 허용된 로그인 ID 목록으로 판정해야 한다.
- 관리자 페이지는 로그인 후 관리자 계정만 접근 가능해야 한다.
- 관리자 화면에서는 다음 기능을 제공해야 한다.
  - 전체 게임 목록 조회
  - 검색
  - 빠른 필터
    - 전체
    - 공개
    - 숨김
    - 신고 alert
    - 플래그
    - 삭제됨
  - 숨김
  - 숨김 해제
  - 삭제
  - 새로고침
- 관리자 페이지 상단은 더 심플해야 하며, `Admin` 배지와 불필요한 설명 카드는 제거해야 한다.
- 관리자 화면은 관리자용 도구 페이지답게 불필요한 설명 문구를 최소화해야 한다.
- 관리자 화면에서 다음 정보를 보여줘야 한다.
  - 썸네일
  - 제목, 설명
  - 작성자
  - 게임 ID
  - 생성/수정일
  - 신고 수
  - 좋아요/싫어요
  - 플레이 수
  - 숨김 사유
  - 리더보드 사용 여부
  - allowlist 위반 여부
  - 광고/포인트 관련 운영 상태를 확장할 수 있는 구조

### 4.18 버그 및 UI/UX 개선 요구

- AI 생성과 AI 수정은 동일한 수준의 진행상황 경험을 제공해야 한다.
- AI 생성과 AI 수정은 모두 모델 선택, 비용 안내, 잔액 안내를 제공해야 한다.
- 게임 상세의 공개 액션 영역은 좋아요/싫어요/피드백/신고가 명확히 구분되어야 한다.
- 사용자의 기존 반응 상태가 새로고침 후에도 버튼에 반영되어야 한다.
- 게임 생성/수정 완료 후 즉시 다른 화면으로 이동하더라도 완료 메시지나 결과 상태를 인지할 수 있어야 한다.
- 광고는 게임 조작 UI, 전체화면 버튼, 리더보드 입력 흐름을 가리면 안 된다.
- 포인트가 필요한 행동에서는 예상 차감량과 잔액 부족 여부가 사전에 보여야 한다.
- 생성 화면과 수정 화면의 AI 모델 선택 UX는 같은 문체, 같은 정보량, 같은 시각 체계를 유지해야 한다.
- 홈, 로그인, 생성, 수정, 관리자, 포인트, 내 게임 화면 전반에서 텍스트 깨짐 없이 한국어/영어가 일관되게 보여야 한다.

## 5. 비기능 요구사항

### 5.1 저장소/운영 모드

- 서비스는 `supabase` 모드만 지원해야 한다.
- 게임 메타데이터는 Supabase에 저장해야 한다.
- 게임 자산은 R2에 저장해야 한다.
- filesystem 기반 저장소 로직은 운영 대상에서 제외해야 한다.

### 5.2 보안

- 게임 자산 경로는 상위 디렉터리 접근을 허용하면 안 된다.
- HTML 게임 응답에는 제한된 CSP를 적용해야 한다.
- 외부 URL은 허용된 CDN 도메인만 정상 범위로 간주해야 한다.
- 세션/반응/플레이 쿠키는 `httpOnly`, `sameSite=lax` 정책을 사용해야 한다.
- IP 정보는 저장 시 직접값 대신 해시 또는 그에 준하는 안전한 방식으로 다뤄야 한다.
- 포인트 차감과 지급은 서버가 최종 권한을 가져야 한다.

### 5.3 성능/운영

- 홈, 게임 상세, 내 게임, 관리자 화면은 동적 렌더링을 허용해야 한다.
- 게임 생성/수정/관리 작업 후 관련 경로는 재검증되어야 한다.
- 광고 SDK가 로드되더라도 게임 플레이 성능이 급격히 저하되지 않아야 한다.
- 홈 화면 리더보드의 자동 이동과 수동 드래그는 저사양 모바일 환경에서도 자연스럽게 동작해야 한다.

### 5.4 정책/컴플라이언스

- 아동 대상 서비스에 맞는 광고 정책을 준수해야 한다.
- 개인화 광고 금지, child-directed 처리, 보상형 광고 opt-in 원칙을 문서화해야 한다.
- 포인트 구매 기능은 결제 및 아동 보호 정책 검토를 선행해야 한다.

## 6. 품질 및 정비 요구

- 현재 확실히 사용하지 않는 소스와 레거시 엔드포인트는 정리 대상이어야 한다.
- AI 생성 경로는 하나의 표준 경로와 하나의 표준 구현으로 정리되어야 한다.
- 인코딩이 깨진 한국어 문구는 모두 정상 문구로 복구되어야 한다.
- 현재 정책과 맞지 않는 fallback 또는 미사용 보안 로직은 유지 목적이 명확하지 않으면 제거해야 한다.
- 버그와 UX 개선 항목은 신규 기능과 별도로 식별되고 우선순위가 부여되어야 한다.

## 7. 요구사항 우선 정리

현재 기준 핵심 요구사항은 다음 다섯 가지로 요약된다.

1. 아이들이 쉽게 게임을 만들 수 있는 간단한 제작 경험 제공
2. 초안-테스트-게시-숨김 흐름을 통해 안전하게 공개할 수 있는 운영 구조 제공
3. 공개 게임 플레이, 반응, 리더보드, 관리자 검수를 포함한 커뮤니티 운영 기능 제공
4. 광고와 포인트를 결합한 지속 가능한 경제 시스템 구축
5. AI 모델 선택, 포인트 비용, 진행상황 표시와 쉬운 문구를 포함한 명확한 AI 제작 UX 제공
